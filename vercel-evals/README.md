## Setup

1. **Install dependencies:**

   ```bash
   pnpm install
   ```

2. **Configure environment variables:**

   ```bash
   cp .env.example .env.local
   ```

   Edit `.env.local` and add your API keys (see comments in `.env.example` for options):
   - **Choose ONE agent key**: `AI_GATEWAY_API_KEY` (for Vercel agents), `ANTHROPIC_API_KEY`, or `OPENAI_API_KEY`
   - **Choose ONE sandbox option**: `VERCEL_TOKEN`, `VERCEL_OIDC_TOKEN`, or use Docker (set `sandbox: 'docker'` in config)

## Running Evals

### Preview (no cost)

See what will run without making API calls:

```bash
npx @vercel/agent-eval claude --dry
```

### Run Experiments

Run the Claude Code Storybook plugin eval fixtures:

```bash
npx @vercel/agent-eval claude
```

Run the Codex Storybook plugin eval fixtures:

```bash
npx @vercel/agent-eval codex
```

Each experiment is one agent. Each eval fixture owns one scenario:

- `922-skill-storybook-setup-claude-launch`: configure `.claude/launch.json` for Storybook
- `923-skill-stories`: change the Badge component, write stories, and preview them

The Claude experiment runs both fixtures. The Codex experiment runs `923-skill-stories`;
`922-skill-storybook-setup-claude-launch` is specific to Claude's launch config.

The Codex experiment uses the Vercel AI Gateway Codex adapter
(`agent: 'vercel-ai-gateway/codex'`), so it uses the same gateway credential
path as the Claude experiment.

#### Number of runs

Each fixture runs once by default. Set `EVAL_RUNS` to run each fixture multiple times:

```bash
EVAL_RUNS=3 npx @vercel/agent-eval claude
```

The eval CI workflow exposes this as a `runs` input on manual (`workflow_dispatch`)
runs and defaults to 1 otherwise.

### Skill injection

The Storybook plugin skills are **not** committed into the eval fixtures. Each
experiment's `setup` hook injects them into the sandbox via `lib/skills-fixture.ts`,
which reads the canonical skills from the plugin packages and applies a single
eval-only overlay to the `stories` skill: a sandbox `require_escalated` note (the
sandbox may otherwise block localhost access for Storybook CLI commands).

The skill tells the agent to "open the Storybook preview in the preview browser" â€” a
native Claude/Codex capability the Vercel Agent Eval harness does not provide. Rather
than rewrite the skill, `setup` mocks that capability with a small stdio MCP server
(see [Preview-browser MCP mock](#preview-browser-mcp-mock)), so the environment
resembles production and the unmodified skill works as-is.

Skill **invocation** is detected from the transcript â€” Claude's `Skill` tool call,
or Codex reading the skill's `SKILL.md` â€” so the skill content is otherwise
unmodified (no injected marker).

| Sandbox dir      | Canonical source                                 |
| ---------------- | ------------------------------------------------ |
| `.claude/skills` | `packages/claude-plugin/skills`                  |
| `.agents/skills` | `packages/codex-plugin/plugins/storybook/skills` |

The plugin packages stay the single source of truth for skill content â€” update a
skill in `packages/`, not in a fixture. Because the content is read live at setup,
refreshing a skill there changes what the eval exercises, so re-run the experiments
after a skill change.

### Validation

`922` and `923` are **strict evals**: each fixture ships an `EVAL.ts` that asserts on the
agent's result with vitest (hard pass/fail). Experiments run with `validation: 'vitest'`,
so after the agent finishes the harness runs `EVAL.ts` in the sandbox alongside the
`build` script (`tsc --noEmit`); the fixture fails if any assertion or the build fails.

The harness writes two files under `__agent_eval__/` before validation: `results.json`
(`o11y` â€” the transcript summary: shell commands, file ops) and `transcript.txt` (the
raw agent transcript). The summary aggregates tool-call args away, so for the
preview-open `923/EVAL.ts` parses `transcript.txt` itself â€” finding a browser-named tool
*call* whose args carry a Storybook URL (a tool definition/mention doesn't count). The
agent can also read the files it generated directly. The strict evals assert:

- **`922-skill-storybook-setup-claude-launch`** â€” `.claude/launch.json` gains a
  `Storybook` entry with `autoPort: true`, and the existing `App dev server` entry is
  preserved.
- **`923-skill-stories`** â€” a `*.stories.*` file is written under `src/`, the
  `storybook ai` CLI was run (`o11y.shellCommands`), and the Storybook preview was opened
  via the preview-browser tool with a `:6006` URL (parsed from `transcript.txt`).

### Scoring system (for migrating legacy `eval/` tasks)

`lib/scoring` is a separate, retained mechanism for porting the older `eval/` tasks. It
scores a weighted rubric host-side in `onRunComplete`, which can see the full transcript
(browser URLs, skill invocations) that `EVAL.ts` cannot. It has **no registered scorers
today** â€” `922`/`923` use `EVAL.ts` instead â€” but the infrastructure stays so legacy
tasks can be migrated incrementally.

To migrate a legacy eval with a scorer:

1. Add the fixture under `evals/<fixture-name>` with an `EVAL.ts` (the harness requires
   the file to exist; `export {}` is enough when the rubric is the only gate).
2. Add a scorer in `lib/scoring/scorers/<fixture-name>.ts` via `defineScorer`, including
   a `threshold`.
3. Register it in `lib/scoring/registry.ts` and add coverage to
   `lib/evaluation-scoring.test.ts`.

Scorers receive `{ fixtureName, runData, analysis, agent }` and return itemized weighted
metrics via `totalScore`/`binaryItem`; shared evidence helpers (generated files, shell
commands, skill invocations, Claude launch parsing) live in `lib/scoring/evidence.ts`.
`pnpm run export-results` averages each registered scorer's runs into the
[Results](#results) table and exits non-zero if a mean is below its `threshold`
(`EVAL_SCORE_THRESHOLD` overrides).

```ts
import { defineScorer, binaryItem, totalScore } from '../types.ts';

export const myFixtureScorer = defineScorer({
	fixtureName: 'my-fixture',
	threshold: 70,
	score({ runData, analysis, agent }) {
		return totalScore([binaryItem('did-work', 'Completed the expected workflow', 1, true)]);
	},
});
```

#### Preview-browser MCP mock

`setup` injects a dependency-free stdio MCP server
(`.agent-eval/mcp/preview-browser-server.mjs`, exposing an `open_preview_browser` tool)
and registers it with each agent â€” Claude via a project `.mcp.json`, Codex via a
`[mcp_servers.preview-browser]` stanza appended to `~/.codex/config.toml`. It stands in
for the native preview browser the sandbox lacks. `923`'s `EVAL.ts` asserts the agent
called it with a `:6006` Storybook URL â€” parsed from the raw `transcript.txt` the harness
materializes (a browser-named tool call carrying the URL), so no agent-eval patch is
needed for the preview signal. The host-side scoring system can also consume it via
`workflow.browserUrls`.
