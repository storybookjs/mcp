## Setup

1. **Install dependencies:**

   ```bash
   pnpm install
   ```

2. **Configure environment variables:**

   ```bash
   cp .env.example .env.local
   ```

   Edit `.env.local` and add your API keys (see comments in `.env.example` for options):
   - **Choose ONE agent key**: `AI_GATEWAY_API_KEY` (for Vercel agents), `ANTHROPIC_API_KEY`, or `OPENAI_API_KEY`
   - **Choose ONE sandbox option**: `VERCEL_TOKEN`, `VERCEL_OIDC_TOKEN`, or use Docker (set `sandbox: 'docker'` in config)

## Running Evals

### Preview (no cost)

See what will run without making API calls:

```bash
npx @vercel/agent-eval claude --dry
```

### Run Experiments

Run the Claude Code Storybook plugin eval fixtures:

```bash
npx @vercel/agent-eval claude
```

Run the Codex Storybook plugin eval fixtures:

```bash
npx @vercel/agent-eval codex
```

Each experiment is one agent. Each eval fixture owns one scenario:

- `922-skill-storybook-setup-claude-launch`: configure `.claude/launch.json` for Storybook
- `923-skill-stories`: change the Badge component, write stories, and preview them

The Claude experiment runs both fixtures. The Codex experiment runs `923-skill-stories`;
`922-skill-storybook-setup-claude-launch` is specific to Claude's launch config.

The Codex experiment uses the Vercel AI Gateway Codex adapter
(`agent: 'vercel-ai-gateway/codex'`), so it uses the same gateway credential
path as the Claude experiment.

#### Number of runs

Each fixture runs once by default. Set `EVAL_RUNS` to run each fixture multiple
times so the [Results](#results) success rate is a mean over several runs rather
than a single run's score:

```bash
EVAL_RUNS=3 npx @vercel/agent-eval claude
```

The eval CI workflow exposes this as a `runs` input on manual (`workflow_dispatch`)
runs and defaults to 1 otherwise.

### Skill injection

The Storybook plugin skills are **not** committed into the eval fixtures. Each
experiment's `setup` hook injects them into the sandbox via `lib/skills-fixture.ts`,
which reads the canonical skills from the plugin packages and applies two
eval-only overlays to the `stories` skill:

- a sandbox `require_escalated` note
- an eval preview-browser mock, because the Vercel Agent Eval CLI harness does not
  expose the real Claude/Codex preview browser

Skill **invocation** is detected from the transcript â€” Claude's `Skill` tool call,
or Codex reading the skill's `SKILL.md` â€” so the skill content is otherwise
unmodified (no injected marker).

| Sandbox dir      | Canonical source                                 |
| ---------------- | ------------------------------------------------ |
| `.claude/skills` | `packages/claude-plugin/skills`                  |
| `.agents/skills` | `packages/codex-plugin/plugins/storybook/skills` |

The plugin packages stay the single source of truth for skill content â€” update a
skill in `packages/`, not in a fixture. Because the content is read live at setup,
refreshing a skill there changes what the eval exercises, so re-run the experiments
after a skill change.

### Scoring

Success is the **mean weighted score across runs**, not a per-run pass/fail. Each run
is scored by the `lib/scoring` rubric in `onRunComplete`, which writes the percentage
(and the fixture's `threshold`) to `result.json` at `analysis.evaluation`. Experiments
run with `validation: 'none'` and `earlyExit: false`, so every run executes and feeds
the mean. `EVAL.ts` is a stub â€” both correctness and quality live in the rubric; the
harness only requires the file to exist.

`pnpm run export-results` averages each fixture's runs into a mean score â€” the success
rate in [Results](#results) â€” and **exits non-zero if any fixture's mean is below its
threshold**, which fails CI. The bar is on the aggregate, not on individual runs.

`922-skill-storybook-setup-claude-launch`:

| Metric                                       | Weight |
| -------------------------------------------- | ------ |
| Storybook launch entry with `autoPort: true` | 70 %   |
| Preserves the existing App dev server entry  | 30 %   |

`923-skill-stories`:

| Metric                                                     | Weight |
| ---------------------------------------------------------- | ------ |
| Loaded story rules via the `storybook ai` CLI              | 30 %   |
| Invoked the `stories` skill                                | 20 %   |
| Wrote a `*.stories.*` file                                 | 20 %   |
| Started Storybook and opened the eval preview-browser mock | 30 %   |

The preview-browser metric is intentionally a test double. The injected stories
skill asks agents to run:

```bash
npm run storybook -- --port 6006
node .agent-eval/bin/open-preview-browser.mjs <storybook-preview-url>
```

That command writes `.agent-eval/preview-browser.json`, which the scorer reads
(source, `opened` status, and a well-formed `:6006` preview URL) rather than
fetching a live server â€” the eval harness runs headless and keeps no Storybook
server alive into the validation phase, so the marker is the durable signal.

#### Pass thresholds

Each scorer declares a `threshold` (percent) that the fixture's **mean** score must
meet to pass:

| Fixture                                   | Threshold |
| ----------------------------------------- | --------- |
| `922-skill-storybook-setup-claude-launch` | 100 %     |
| `923-skill-stories`                       | 70 %      |

Tune a fixture's bar by editing `threshold` on its scorer. To override all fixtures
for a single run (e.g. CI tuning), set `EVAL_SCORE_THRESHOLD`; it takes precedence
over the scorer defaults:

```bash
EVAL_SCORE_THRESHOLD=80 npx @vercel/agent-eval claude
```

Export the latest results, including weighted scoring, with:

```bash
pnpm run export-results
```

This writes `agent-results.json` and refreshes the [Results](#results) table at the
top of this README (both the success rate and the weighted score for each fixture).

To migrate scoring for another former eval:

1. Add the fixture under `evals/<fixture-name>` (a stub `EVAL.ts` is enough â€” the
   harness only checks the file exists).
2. Add a scorer in `lib/scoring/scorers/<fixture-name>.ts` with `defineScorer`,
   including a `threshold`. Put every check here â€” the rubric is the only gate.
3. Register it in `lib/scoring/registry.ts`.
4. Add or update `lib/evaluation-scoring.test.ts`.

Scorers receive `{ fixtureName, runData, analysis, agent }` and return itemized
weighted metrics. Shared evidence helpers for generated files, shell commands,
skill invocation markers, and Claude launch config parsing live in
`lib/scoring/evidence.ts`.

```ts
import { defineScorer, binaryItem, totalScore } from '../types.ts';

export const myFixtureScorer = defineScorer({
	fixtureName: 'my-fixture',
	threshold: 70,
	score({ runData, analysis, agent }) {
		return totalScore([binaryItem('did-work', 'Completed the expected workflow', 1, true)]);
	},
});
```
